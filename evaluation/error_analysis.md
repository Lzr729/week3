# Error Analysis

## 1. 哪些事件可以由规则稳定提取

当前规则在以下场景中表现相对稳定：

- 单一认购方现金增资；
- 正文明确列示注册资本前后值的增资；
- 多参与方且每名参与方金额在结构清晰表格或连续句式中列示；
- 股票定向发行中的发行数量、价格、募集资金和资本公积；
- 吸收合并导致的注册资本增加；
- 明确列示转让方、受让方、比例和零对价的内部重组；
- 标准“股东名称—持股数量—持股比例”或“出资额—出资比例”表格。

开发样本中，单一认购方增资、多参与方增资、定向发行和吸收合并增资经过字段Gold回归后表现较好。

## 2. 哪些事件需要LLM或人工复核

以下场景需要人工复核，未来可考虑由LLM辅助：

- 多批次增资被写在同一小节；
- 同一协议同时包括增资与股份转让；
- 多名参与方采用不同价格；
- 最终数值需要跨两页以上汇总；
- PDF文本层将表格列顺序打乱；
- 仅有股权关系图、没有可解析表格；
- 公司简称可能指向发行人、股东或被合并方；
- 披露口径在注册资本、股份数量和实收资本之间切换。

## 3. 自动流程相对人工Gold的主要漏抽

主要漏抽包括：

- 多名参与方的完整列表；
- 每名参与方的认缴注册资本与出资金额；
- 多阶段事件的最终注册资本；
- 跨页股权结构表；
- 股本前值、后值及新增股份；
- 未直接披露但可通过明确等式计算的字段。

云汉芯城最终盲测中，规则只命中第一批或第一名参与方，未完成整个事件的汇总，属于典型漏抽。

## 4. 自动流程相对人工Gold的主要误抽

主要误抽包括：

- 将章节内其他股东的历史出资额识别为本次交易对价；
- 将董事会日期、前次协议日期或清税日期识别为本次关键日期；
- 将上层企业或其他主体的股权变化误认为发行人股权变化；
- 将第一阶段的注册资本误认为整个事件的最终注册资本；
- 将金额和动作文字并入参与方名称。

## 5. 单位错误如何发现

统一保存：

- `raw_text`；
- `value`；
- `unit`；
- `normalized_value`；
- `normalized_unit`。

归一化规则示例：

- `1万元 = 10,000 CNY`；
- `1亿元 = 100,000,000 CNY`；
- `1万股 = 10,000 share`；
- 百分比统一为`percent`。

单位问题通过三层发现：

1. Schema限制允许的单位；
2. Cross-check要求参与等式的字段具有相同`normalized_unit`；
3. Gold比较使用归一化值而不是只比较原始字符串。

## 6. 如何区分股权转让与增资认缴

`subscription_flow`判断依据：

- 注册资本或总股本增加；
- 存在新增注册资本、新增股份、认缴、发行或募集资金；
- 资金流入发行人。

`share_transfer_flow`判断依据：

- 存量股权在转让方与受让方之间移动；
- 公司总注册资本或总股本原则上不变；
- 对价支付给转让方，而不是形成发行人新增资本；
- 必须确认转让标的是发行人或发行人前身。

同一小节同时存在两类事件时，分别生成不同事件包或在人工复核中拆分。

## 7. Cross-check失败后的处理

Cross-check保留：

- expected；
- actual；
- difference；
- normalized_unit；
- check_name；
- package_id；
- status。

处理规则：

- `PASS`：数值等式成立；
- `FAIL`：进入人工复核队列，检查事件切分、单位和字段选择；
- `SKIP`：缺少完成等式所需字段，不视为通过或失败，但保留缺失原因；
- Schema无效：优先级高于一般Cross-check失败。

当前3个FAIL集中于云汉芯城复杂多阶段增资和星图测控股票发行字段口径。

## 8. 扩展到50家公司最先会失败在哪里

最先出现瓶颈的环节预计是：

1. PDF文本层质量差异；
2. 章节标题表达不统一；
3. 跨页复杂表格；
4. 同节多事件切分；
5. 公司简称和主体关系解析；
6. 单位与口径切换；
7. 人工复核队列规模快速增长。

扩展方案：

- 增加PDF解析fallback；
- 将事件包切分和字段抽取分开评估；
- 对表格使用专门解析器；
- 引入主体别名字典；
- 对Schema或Cross-check失败记录自动回源；
- 用LLM只处理规则失败事件，而不是替代全部规则；
- 对人工修正保留diff和原因，持续扩展回归集。

## 9. 本次结果的限制

最终结果证明8家公司和49个候选事件包没有静默遗漏，但不能将覆盖率解释为字段准确率。48条记录仍需人工复核，9条记录JSON Schema无效，189项数值检查中159项因字段不足而SKIP。

因此，本次提交属于可复现的基础闭环，而不是完全自动、无需复核的生产系统。
